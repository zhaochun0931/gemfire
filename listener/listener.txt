1. CacheListener (The Synchronous Watcher)
This is the most standard listener. It is attached to a specific Region (table) and fires synchronously immediately after an operation (like a put or destroy) happens.

How it works: Because it is synchronous, the thread doing the put operation is blocked until the CacheListener finishes its work.

When to use it: Lightweight, fast tasks. For example, triggering a simple alert, logging an event, or updating a secondary local data structure.

Warning: Never put heavy network calls or slow database writes inside a CacheListener. It will drastically slow down your entire GemFire cluster's performance.

2. AsyncEventListener (The Heavy Lifter)
This is the enterprise solution for reacting to data changes without slowing down your cache.

How it works: Instead of firing synchronously, GemFire drops the event into an AsyncEventQueue (AEQ). The AsyncEventListener then picks up these events in batches on a separate background thread.

When to use it: This is perfect for the "Write-Behind" caching pattern. If you want every put in GemFire to eventually be saved to a slow backend database (like Oracle or Postgres), or if you want to publish the event to Kafka, you use an AsyncEventListener.

3. Continuous Query (CqListener)
This one is incredibly powerful and runs on the Client side rather than the Server side.

How it works: You give the cluster an OQL query (e.g., SELECT * FROM /Stocks WHERE price > 100). The server monitors all data changes. Whenever a piece of data changes in a way that makes it match your query, the server pushes that specific event down to the CqListener on your Java client.

When to use it: Real-time dashboards, trading screens, or user interfaces that need to update instantly when specific criteria are met.

4. CacheWriter (The Gatekeeper)
While technically named a "Writer," this acts like a synchronous "Before" listener.

How it works: It fires before the data is actually committed to the GemFire cache.

When to use it: It is used as a gatekeeper. A CacheWriter can do a synchronous check against a backend database (the "Write-Through" pattern) or completely veto/abort the operation by throwing a CacheWriterException if the data is invalid.
